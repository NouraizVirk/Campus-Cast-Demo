## 📌 Project Overview

**CampusCast** is an enterprise-grade, high-trust voting platform engineered specifically for academic institutions. It replaces vulnerable, manual voting mechanisms with a streamlined digital experience, guaranteeing **"one identity, one vote"** through cryptographic integrity. 

Designed with a **Multi-Tenant Architecture**, a single deployment of CampusCast securely serves multiple independent universities, colleges, or distinct departmental branches.

## 🛠️ Technical Stack & Architecture

This application was developed using a modern, scalable JavaScript ecosystem:

### Backend & Infrastructure
- **Node.js & Express.js:** Robust RESTful API and server-side routing.
- **MongoDB & Mongoose (Atlas):** NoSQL database utilized for flexible, scalable document storage.
- **Serverless Edge Deployment:** Configured for high-availability deployment on **Vercel** with optimized `vercel.json` and proxy-trust headers to ensure strict cookie security over serverless topologies.
- **Connect-Mongo:** Persisted, distributed session storage.

### Security & Authentication
- **Passport.js & bcryptjs:** Implementation of robust session-based authentication and secure password hashing.
- **Role-Based Access Control (RBAC):** Custom middleware strictly isolates routing and data access based on three distinct hierarchical roles.
- **Cryptographic Voting Verification:** Prevents ballot-stuffing and race conditions during high-concurrency election events.
- **Helmet.js:** Hardened HTTP headers and Content Security Policies (CSP) to mitigate XSS and clickjacking attacks.

### Frontend Engineering
- **Server-Side Rendering (SSR):** Utilizes **EJS** for ultra-fast initial page loads and secure data hydration without client-side vulnerabilities.
- **Tailwind CSS:** Fully responsive, utility-first UI using modern CSS variables, CSS grid/flexbox, and "Bento Grid" dashboard layouts.
- **Material Symbols:** Integrated for lightweight, dynamic iconography.
- **Dynamic Theming:** Adheres to modern "Institutional Minimalism" with customized, accessible color palettes.

## 💡 Engineering Highlights

### 1. Multi-Tenant Data Isolation
To allow a single application instance to serve various universities, the database implements an `Institute` and `Enrollment` model. 
- Organizers are dynamically linked to specific Institutes securely via a global `AUTH_CODE` generated by the Super Admin. 
- Student users create linkage requests to their respective institutions, which must be explicitly approved by an authorized Organizer before they are permitted to vote.

### 2. Immutable Audit Logging
Transparency is critical in election software. All significant administrative actions (user approval, election creation, setting modifications) automatically trigger immutable entries in a global `AuditLog` collection, accessible exclusively by the Super Admin.

### 3. Serverless Session Management
Adapted standard Express session management to operate flawlessly on Vercel's ephemeral serverless architecture. This included proxy-trust resolution and `MongoDBStore` integrations to preserve user sessions across cold starts and edge node requests.

## 👥 User Roles & Workflows

1. **👑 Super Admin**
   - System-wide visibility.
   - Responsible for registering new Institutes and managing Organizer authorization codes.
   - Monitors the real-time global Audit Log.
2. **🎓 Organizer (Institute Admin)**
   - Manages localized elections within their designated Institute.
   - Approves/rejects student enrollments.
   - Adds voting positions, candidates, and monitors cryptographic tallies.
3. **👤 Student (Voter)**
   - Enrolls in their respective Institute.
   - Can view only active elections corresponding to their approved status.
   - Casts secure, anonymous votes with real-time feedback.
